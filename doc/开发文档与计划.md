# AI-agent-TODO 4 人团队开发文档与开发计划

## 1. 文档目的

本文档基于 `doc/项目初步设计.md` 与 README 中的项目定位，对 **AI-agent-TODO 智能任务管理系统** 进行开发落地拆解，明确 4 人团队的职责边界、功能模块、技术路线、阶段计划、交付物与协作规范。

本文档适用于课程设计、团队开发启动、迭代排期、阶段汇报和项目验收准备。

## 2. 项目概述

AI-agent-TODO 是一个以 Web 端为核心的智能任务管理系统，面向个人用户提供任务创建、查看、编辑、删除、完成状态管理、分类、优先级、截止时间、统计分析等基础 TODO 能力，并引入 AI Agent 能力，支持自然语言解析任务、智能分类和优先级推荐。

系统采用 **BYOK（Bring Your Own Key）** 模式：用户在个人设置中配置自己的 OpenAI API Key，系统后端使用该 Key 调用 AI 能力，从而降低统一部署成本并提升项目可扩展性。

## 3. 项目目标与范围

### 3.1 核心目标

1. 完成一个可运行的 Web 端智能 TODO 管理系统。
2. 支持用户注册、登录和个人任务数据隔离。
3. 支持任务的增删改查、完成状态切换、分类、优先级和截止时间管理。
4. 支持 AI 自然语言解析任务，并自动推荐分类与优先级。
5. 支持任务统计分析，包括完成率、待办数量和分类统计。
6. 形成完整的软件工程材料，包括需求、设计、测试、部署与总结文档。

### 3.2 第一阶段范围（MVP）

MVP 版本优先实现以下能力：

- 用户注册、登录、退出登录。
- 任务列表、任务详情、任务创建、任务编辑、任务删除。
- 任务完成 / 未完成状态切换。
- 任务优先级、分类、截止时间字段。
- AI 自然语言创建任务。
- 用户 OpenAI API Key 配置。
- 简单统计面板。
- 前后端联调与本地部署说明。

### 3.3 暂不纳入 MVP 的范围

以下能力作为后续扩展方向，不作为第一阶段验收的强制要求：

- 移动端 App。
- 桌面端应用。
- 多人协作任务空间。
- 多模型供应商切换。
- 日历双向同步。
- 复杂提醒推送系统。

## 4. 技术架构设计

### 4.1 总体架构

```text
React + Vite 前端
        ↓ HTTP/JSON
FastAPI 后端服务
        ↓ ORM
SQLite / PostgreSQL 数据库
        ↓ 用户 BYOK
OpenAI API
```

### 4.2 前端技术方案

- 技术栈：React、Vite、TypeScript、Axios、React Router。
- 状态管理：MVP 可优先使用 React Hooks；如复杂度提升，可引入 Zustand 或 Redux Toolkit。
- 页面模块：登录注册页、任务列表页、任务编辑页、AI 创建任务页、统计页、设置页。
- 交互重点：任务快速创建、AI 解析结果确认、列表筛选与排序、错误提示。

### 4.3 后端技术方案

- 技术栈：FastAPI、Pydantic、SQLAlchemy、Uvicorn。
- 认证方式：JWT Token。
- 数据库：开发阶段使用 SQLite，后续部署可切换 PostgreSQL。
- API 风格：RESTful API。
- 重点模块：用户认证、任务管理、AI 服务、统计服务、配置管理。

### 4.4 AI 模块技术方案

- 接入方式：OpenAI API。
- Key 管理：用户在设置页配置 API Key，后端加密或脱敏保存。
- 核心能力：自然语言任务解析、分类推荐、优先级推荐。
- 输出格式：后端要求 AI 返回结构化 JSON，并进行字段校验和兜底处理。

### 4.5 数据库核心实体

| 实体 | 说明 | 关键字段 |
| --- | --- | --- |
| User | 用户账户 | id、username、email、password_hash、created_at |
| Task | 用户任务 | id、user_id、title、description、priority、category、due_time、status、created_at、updated_at |
| UserSetting | 用户设置 | id、user_id、openai_api_key、model_name、created_at、updated_at |
| AiCallLog | AI 调用记录 | id、user_id、input_text、output_json、status、created_at |

## 5. 功能模块拆解

### 5.1 用户管理模块

**功能说明：**

- 用户注册。
- 用户登录。
- JWT 鉴权。
- 获取当前用户信息。
- 修改基础个人信息。

**主要接口：**

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/auth/register` | 注册用户 |
| POST | `/api/auth/login` | 登录并返回 Token |
| GET | `/api/users/me` | 获取当前用户信息 |
| PUT | `/api/users/me` | 修改当前用户信息 |

### 5.2 任务管理模块

**功能说明：**

- 创建任务。
- 查看任务列表。
- 查看任务详情。
- 修改任务。
- 删除任务。
- 标记任务完成 / 未完成。
- 按分类、优先级、完成状态、截止时间筛选任务。

**主要接口：**

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/tasks` | 创建任务 |
| GET | `/api/tasks` | 获取任务列表 |
| GET | `/api/tasks/{task_id}` | 获取任务详情 |
| PUT | `/api/tasks/{task_id}` | 更新任务 |
| DELETE | `/api/tasks/{task_id}` | 删除任务 |
| PATCH | `/api/tasks/{task_id}/status` | 更新任务状态 |

### 5.3 AI 自然语言解析模块

**功能说明：**

用户输入自然语言任务描述，系统调用 AI 服务解析出结构化任务字段。例如：

```text
输入：明天下午三点完成软件工程报告，很重要
输出：
{
  "title": "完成软件工程报告",
  "due_time": "明天下午三点",
  "priority": "high",
  "category": "学习"
}
```

**主要接口：**

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/ai/parse-task` | 解析自然语言任务 |
| POST | `/api/ai/create-task` | 解析并创建任务 |
| POST | `/api/ai/suggest` | 推荐分类与优先级 |

### 5.4 统计分析模块

**功能说明：**

- 统计任务总数。
- 统计已完成任务数量。
- 统计未完成任务数量。
- 计算任务完成率。
- 按分类统计任务数量。
- 按优先级统计任务数量。

**主要接口：**

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/stats/overview` | 获取任务统计总览 |
| GET | `/api/stats/category` | 获取分类统计 |
| GET | `/api/stats/priority` | 获取优先级统计 |

### 5.5 设置与 BYOK 模块

**功能说明：**

- 保存用户 OpenAI API Key。
- 修改默认模型名称。
- 测试 API Key 是否可用。
- 展示 API Key 脱敏结果。

**主要接口：**

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/settings` | 获取用户设置 |
| PUT | `/api/settings` | 更新用户设置 |
| POST | `/api/settings/test-openai-key` | 测试 API Key |

## 7. 开发计划与里程碑

### 7.1 总体周期

建议开发周期为 **4 周**，每周形成一个可检查的阶段成果。

| 周期 | 阶段目标 | 主要任务 | 阶段交付物 |
| --- | --- | --- | --- |
| 第 1 周 | 需求确认与架构搭建 | 明确功能范围、创建前后端项目、设计数据库、确定接口规范 | 需求文档、数据库设计、前后端脚手架 |
| 第 2 周 | 基础功能开发 | 用户模块、任务 CRUD、前端基础页面、接口联调 | 可登录并管理任务的基础系统 |
| 第 3 周 | AI 与统计功能开发 | BYOK 设置、AI 解析、智能分类、统计看板、异常处理 | 支持 AI 创建任务和统计分析的系统 |
| 第 4 周 | 测试、优化与交付 | 系统测试、Bug 修复、部署说明、答辩材料、项目总结 | 可演示版本、测试报告、完整文档 |

### 7.2 每周详细安排

#### 第 1 周：需求确认与项目初始化

- 成员 A：设计数据库表结构，搭建 FastAPI 项目，定义 API 路由规范。
- 成员 B：搭建 React + Vite 项目，完成页面路由和基础布局。
- 成员 C：整理 AI 功能需求，设计 Prompt 初稿与 JSON 输出规范。
- 成员 D：整理需求分析文档、任务分解表和项目排期表。

**验收标准：**

- 前后端项目可以本地启动。
- 数据库实体和接口清单已确定。
- 文档目录和基础文档已建立。

#### 第 2 周：用户与任务基础功能

- 成员 A：完成用户注册登录、JWT 鉴权、任务 CRUD 接口。
- 成员 B：完成登录注册页、任务列表页、任务编辑弹窗或页面。
- 成员 C：提供 AI 模块 Mock 接口，便于前端提前联调。
- 成员 D：编写基础功能测试用例，开始记录接口测试结果。

**验收标准：**

- 用户可以注册、登录并看到自己的任务列表。
- 用户可以创建、编辑、删除、完成任务。
- 前后端完成基础联调。

#### 第 3 周：AI 功能与统计分析

- 成员 A：完成统计接口和设置接口。
- 成员 B：完成统计页、设置页、AI 创建任务交互。
- 成员 C：接入 OpenAI API，完成自然语言解析、分类推荐和优先级推荐。
- 成员 D：补充 AI 功能测试用例，开展系统集成测试。

**验收标准：**

- 用户可以配置自己的 OpenAI API Key。
- 用户可以通过自然语言生成结构化任务。
- 系统可以展示任务完成率、分类统计和优先级统计。

#### 第 4 周：测试、优化与最终交付

- 成员 A：修复后端 Bug，完善错误处理和数据校验。
- 成员 B：优化页面样式、交互体验和响应式布局。
- 成员 C：优化 Prompt，完善 AI 失败兜底逻辑。
- 成员 D：完成测试报告、部署文档、项目总结和演示脚本。

**验收标准：**

- 系统可稳定完成核心演示流程。
- 关键接口和主要页面通过测试。
- 项目文档齐全，可用于课程提交或项目汇报。

## 8. 协作流程

### 8.1 Git 分支建议

- `main`：稳定版本分支。
- `develop`：集成开发分支。
- `feature/backend-auth`：用户认证功能分支。
- `feature/backend-task`：任务管理功能分支。
- `feature/frontend-pages`：前端页面功能分支。
- `feature/ai-parse-task`：AI 解析功能分支。
- `feature/docs-test-deploy`：文档、测试和部署分支。

### 8.2 提交信息规范

建议使用以下格式：

```text
类型: 简短说明
```

常用类型：

- `feat`：新增功能。
- `fix`：修复问题。
- `docs`：文档变更。
- `test`：测试相关。
- `refactor`：重构。
- `chore`：工程配置或杂项。

示例：

```text
feat: add task CRUD APIs
docs: add deployment guide
fix: handle empty AI parse result
```

### 8.3 联调流程

1. 后端先提供 Swagger 或接口说明。
2. 前端基于 Mock 数据完成页面开发。
3. 后端接口完成后，前端切换真实接口。
4. AI 模块先提供 Mock 输出，再接入真实 OpenAI API。
5. 测试负责人统一记录联调问题和修复状态。

## 10. 风险与应对措施

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| OpenAI API Key 不可用 | AI 功能无法演示 | 提供 Mock 模式和示例输出 |
| 前后端接口不一致 | 联调进度受阻 | 第 1 周冻结接口字段，变更需同步文档 |
| 时间不足 | 部分功能无法完成 | 优先保障 MVP，扩展功能延后 |
| 数据库设计频繁变更 | 影响后端开发 | 先设计核心字段，非必要字段延后 |
| 团队任务边界不清 | 工作重复或遗漏 | 按模块分配责任人，每周检查交付物 |
| AI 输出不稳定 | 创建任务字段错误 | 使用 JSON Schema 校验与默认值兜底 |

## 11. 验收标准

项目最终验收建议从以下方面检查：

1. 用户可以完成注册、登录和退出。
2. 用户任务数据能够按用户隔离。
3. 用户可以完成任务增删改查和状态切换。
4. 用户可以配置 BYOK，并触发 AI 任务解析。
5. AI 解析结果能生成结构化任务字段。
6. 系统能够展示任务统计数据。
7. 前后端能够本地启动并完成完整演示流程。
8. README、需求、设计、测试、部署、总结等文档基本完整。

## 12. 推荐演示流程

1. 打开系统首页，展示项目名称和登录入口。
2. 注册新用户并登录。
3. 创建一条普通任务。
4. 输入自然语言任务：`明天下午三点完成软件工程报告，很重要`。
5. 展示 AI 解析出的标题、截止时间、优先级和分类。
6. 确认创建 AI 任务。
7. 修改任务状态为已完成。
8. 打开统计页面展示完成率、分类统计和优先级统计。
9. 打开设置页面展示 BYOK 配置入口。
10. 总结项目架构、创新点和后续扩展方向。
