# AI-agent-TODO 当前前端风格指南

采样时间：2026-06-05 10:59 左右。  
采样方式：真实浏览器打开 `http://127.0.0.1:5174/`，逐页截图、采集 computed style，并对 `frontend/src/styles/demo.css` 做样式证据分析。  
证据文件：`evidence/computed-styles.json` 与 `evidence/screenshots/`。

## 1. 总体视觉结论

当前前端已经不是普通 Todo List，而是一个深色后台管理 / 产品级 Dashboard 的方向：左侧固定导航、顶部工具栏、Dashboard、任务表格、Kanban、日历、统计、设置等页面都已成型。整体第一屏有 AI TODO 的产品感，尤其是 Dashboard 的 “AI 智能建议”、AI 推荐页、详情抽屉里的 AI 分析结果。

主要问题是风格体系仍然混杂：全局 `:root` 保留浅色 token，但 `.minimal-shell` 强制进入深色 UI，导致部分表单、表格按钮、标签仍带浅色变量痕迹；视觉层级有时偏平，AI 区域虽然有内容，但缺少足够稳定的 AI 视觉识别系统。移动端已有底部导航和安全区 padding，但任务表格仍是 900px 横向滚动，app 端阅读体验还不够像原生工具。

已观察页面：Dashboard、全部任务、AI 推荐、任务看板、日历、数据统计、标签管理、设置、新建任务弹窗、任务详情 Drawer、深色模式、移动端布局。  
未观察到独立 “今日任务页面”：当前 “今日任务” 出现在 Dashboard 模块内。  
未观察到可单独截图的编辑任务弹窗和删除确认弹窗：任务表格存在 “编辑任务 / 删除任务” 按钮，但本轮没有形成独立弹窗证据。

## 2. 色彩系统

| 用途 | 颜色值 | 出现场景 | 问题 |
|---|---|---|---|
| 全局浅色背景 token | `#f5f7fb` | `:root --bg` | 与当前深色主界面并存，容易让组件出现浅色遗留 |
| 深色页面背景 | `#09090b` | `.minimal-shell` 主壳 | 方向统一，但面积很大，需要更多层次 |
| 深色卡片背景 | `#121214` | `.content-card`、`.ai-card`、`.drawer`、`.create-modal` 深色覆盖 | 专业安静，但部分模块显得偏扁 |
| 深色内层卡片 | `#18181b` | 任务卡、Kanban 卡、输入框、Drawer 字段 | 层级清楚，可继续作为二级 surface |
| 主文本 | `#cbd5e1` / `#f8fafc` | 正文、标题 | 标题和正文对比可用 |
| 弱化文本 | `#64748b` / `#94a3b8` | 描述、meta、状态说明 | 某些表格正文偏灰，长表格阅读略吃力 |
| 主色 | `#4f46e5` | 新建任务按钮、品牌、AI/active 状态 | 统一度较好 |
| 主色 hover | `#4338ca` | `.minimal-primary:hover` | 有 hover，但动效很轻 |
| Active 导航 | `rgba(79, 70, 229, 0.12)` + `rgb(129, 140, 248)` | Sidebar / mobile nav active | 可识别，但侧栏 active 背景偏弱 |
| 深色边框 | `rgba(255,255,255,0.06)` | 卡片、Sidebar、移动底部导航 | 统一但对比略低 |
| Focus 边框 | `rgba(99,102,241,0.7)` | 搜索框 focus | 只有少数组件明显 |
| 成功状态 | `rgb(22,163,74)` / `#16a34a` | 已完成状态 | 当前主要靠文字色，badge 背景不够强 |
| 进行中/中优先级 | `rgb(180,83,9)` / `rgb(194,65,12)` | 进行中、中优先级 | 色相相近，容易和警告混在一起 |
| 高优先级/错误 | `rgb(220,38,38)` / `#dc2626` | 高优先级、错误 token | 醒目，但 badge 视觉面积小 |
| AI 区域 | `#4f46e5`、`#7c3aed`、透明渐变 | AI 推荐、AI 分析、趋势图 | 有 AI 气质，但分散在多个模块里，缺一个统一 AI accent 规则 |

## 3. 字体系统

| 类型 | 字体 | 字号 | 字重 | 行高 | 使用场景 |
|---|---|---|---|---|---|
| 全局字体 | `Inter, ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", "PingFang SC", "Microsoft YaHei", sans-serif` | 继承 | 继承 | `1.5` | 全局与 `.minimal-shell` |
| Dashboard H1 | 同上 | 约 32px 级别 | 800-900 | 约 1.15-1.25 | 首页主问候语 |
| 页面标题 | 同上 | 24-30px 区间 | 800-900 | 紧凑 | `全部任务`、`AI 推荐`、`数据统计` |
| 卡片标题 | 同上 | 16-18px | 800-900 | 约 1.4 | AI 卡片、统计/分布卡 |
| 正文 | 同上 | 14px | 400-600 | 1.5 | 任务描述、表格正文 |
| 辅助文字 | 同上 | 12-13px | 600-800 | 1.4 | meta、标签、说明 |
| 按钮文字 | 同上 | 14px / 移动底部 11px | 800-900 | 1 | 主按钮、导航、分段控件 |
| Badge 文字 | 同上 | 12-13px | 800-900 | 24px 高度内居中 | 状态和优先级 |

当前文字系统的优点是足够克制，适合工具型应用。问题是字重偏重的地方很多，标题、按钮、badge、表头都偏 800-900，导致界面层级靠粗细区分不够细腻。建议后续保留标题强度，但把表格、过滤控件、标签的字重降一档。

## 4. 间距系统

| 场景 | 当前值 | 是否统一 | 问题 |
|---|---|---|---|
| Sidebar 宽度 | `240px` | 统一 | 桌面清晰 |
| Header 高度 | `64px` | 统一 | 移动端换行后高度变高 |
| 页面主内容 padding | 桌面 `24px 32px 32px`，移动 `18px 16px 28px` | 基本统一 | 移动端底部导航遮挡需继续留安全区 |
| Dashboard 最大宽度 | `1120px`，居中 | 统一 | 宽屏不会过度拉伸 |
| Dashboard 卡片 gap | `32px` | 统一 | 信息块间距偏舒展 |
| 内容页卡片 padding | `20px` 或 `24px` | 基本统一 | 不同页面密度略不一致 |
| 统计卡片 padding | `16px` / `18px` | 基本统一 | 可以统一成 16/20 标尺 |
| 表单控件高度 | `42px` | 统一 | 桌面可用，移动端可加大点击区 |
| 按钮内边距 | 主按钮 `0 16px`，移动 `0 12px` | 统一 | 主按钮视觉清楚 |
| Kanban 列 gap | `16px` | 统一 | 桌面好用，1180px 以下改为单列 |
| 日历日卡 padding | `14px` | 统一 | 7 日视图尚可，14/30 天会拥挤 |

## 5. 布局系统

桌面布局是 `.minimal-shell` 两列 grid：`240px minmax(0, 1fr)`，左侧 Sidebar，右侧 `.minimal-workspace` 包含 Header 和可滚动 `.minimal-page`。Dashboard 内容居中限制在 1120px，其他功能页多数使用 1136px 左右内容宽度。

移动端在 `max-width: 900px` 下变为单列：Sidebar 只保留顶部品牌条，`.minimal-nav` 隐藏，`.minimal-mobile-nav` 固定底部，宽度 100%，高度约 69px，8 个 tab 横向滚动。这个方向适合 app 套壳网页，但需要把任务表格、抽屉、弹窗继续改成更 app 化的卡片/底部 sheet。

## 6. 组件风格

### Sidebar

桌面宽度 240px，背景跟主壳一致为深色，右侧边框 `rgba(255,255,255,0.06)`。菜单项高度 42px，左右 padding 12px，图标来自 lucide 风格，active 使用淡紫背景和 `rgb(129,140,248)` 文字。当前没有明显分组标题，只是线性导航。移动端 Sidebar 导航隐藏，转为底部 tab。

### Header

Header 高 64px，padding `0 32px`，左侧搜索框宽 320px，右侧包含新建任务、通知、主题切换、模式状态、用户、退出。搜索框是 999px 胶囊，背景 `#121214`，focus 边框变紫。移动端 Header 会换行，搜索框占满第二行，图标按钮隐藏，用户 pill 宽度压缩到 104px。

### TaskCard

当前任务在不同页面有三种呈现：Dashboard 的今日任务行、Kanban 卡、表格行。Kanban 卡是较合理的移动候选形态：深色卡片、14px padding、14px 圆角、8px gap，显示标题、描述、分类、截止时间、AI 分类。表格行适合桌面管理，但移动端仍保持 900px 表格横滚，不适合 app 端。

### TaskTable

全部任务页采用 shadcn Tasks Example 风格：顶部筛选栏，下面固定表格。表格 `min-width: 900px`，列包括任务名称、状态、优先级、分类、截止时间、创建时间、操作。优点是字段完整，缺点是深色壳里还存在浅色变量痕迹，行操作按钮只有图标且密度较高；移动端横向滚动会让操作成本上升。

### TaskBoard

Kanban 是三列：待办、进行中、已完成。桌面 `repeat(3, minmax(0, 1fr))`，gap 16px；1180px 以下变单列。列背景比卡片更轻，任务卡有 AI 分类。整体是当前最适合继续扩展为 app 卡片体验的模块。

### AIAssistantCard

Dashboard 有 AI 智能建议，AI 推荐页有 AI 智能建议、AI 推荐分类和优先级、AI 调用记录，Drawer 有 AI 分析结果。AI 信息真实存在，包括推荐原因、自动分类、预计时间等。视觉上目前主要靠紫色和文案体现 AI，建议后续建立统一 AI surface：图标、淡紫边框、解释原因区、置信度/预计时间小标签。

### CreateTaskModal

新建任务弹窗宽 `min(620px, 100%)`，桌面居中，深色背景下实际为 `#121214`，padding 22px，gap 14px。顶部有 `CREATE TASK` 和标题 “新建任务”，分段控件 `AI 生成 / 自定义创建`，AI 模式下有 textarea、主按钮、说明卡片。移动端 CSS 已做底部对齐和 100% 宽，但需要按 app bottom sheet 继续强化拖拽柄、关闭方式和安全区。

### TaskDetailDrawer

Drawer 是右侧固定 aside，宽 `min(460px, 100vw)`，桌面 1440 下实际 x=980、width=460，高 900，padding 22px。内容包括任务标题/描述、字段网格、子任务、AI 分析结果、底部操作。深色 shell 下可读性不错；移动端宽 100vw，但还不像全屏详情页，建议加顶部安全区、固定操作栏和返回手势语义。

### StatsCard

统计页有 6 个 KPI：任务总数、已完成、待处理、逾期任务、今日截止、AI 创建数量。卡片高度 112px，padding 18px，图标 44px 左侧布局。趋势图是 CSS 柱状图，支持 7/14/30/24h。当前能表达数据，但专业度偏演示稿，图例、坐标、hover tooltip、空状态还可以增强。

## 7. 当前视觉问题

| 问题 | 表现 | 影响 |
|---|---|---|
| 深浅 token 混用 | `:root` 是浅色变量，`.minimal-shell` 再覆盖深色 | 表格、行操作、表单可能出现浅色残留 |
| AI 视觉识别不够系统 | AI 内容多，但样式分散 | 产品特色没有形成稳定记忆点 |
| 表格移动端体验差 | 390px 下表格仍 900px 宽横滚 | app 端不好读、操作按钮难点 |
| Badge 只靠文字色 | 状态和优先级背景弱 | 扫描效率一般 |
| 动效偏少 | 多数 transition 是 `all` 或仅颜色变化 | 交互反馈轻，少了精致感 |
| 删除/编辑弹窗证据不足 | 有入口但没有独立弹窗观察证据 | 后续需要补齐交互链路 |
| 24 小时视图可用但弱 | timeline 已存在，但视觉重点不强 | 不能很好表达一天内任务密度 |

## 8. 可复用风格规则

| 规则 | 建议值 |
|---|---|
| 主背景 | `#09090b` |
| 一级 surface | `#121214` |
| 二级 surface | `#18181b` |
| 主文本 | `#f8fafc` 标题，`#cbd5e1` 正文 |
| 弱文本 | `#94a3b8` 常规弱化，`#64748b` 次级 meta |
| 主色 | `#4f46e5`，hover `#4338ca` |
| 边框 | `rgba(255,255,255,0.06)`，输入 focus `rgba(99,102,241,0.7)` |
| 卡片圆角 | 工具型 UI 建议 12-16px；当前 14/16/18px 可收敛 |
| 按钮圆角 | 主按钮 999px 可保留，表格 icon button 建议 8-10px |
| 页面 padding | 桌面 24/32，移动 16/18 |
| App 底部导航 | `.minimal-mobile-nav` 固定底部，保留 safe-area padding |

## 9. 后续美化方向

1. 先统一深色设计 token，把浅色变量从主应用态中隔离到登录页或 light theme。
2. 全部任务页在移动端改为任务卡片列表，桌面保留表格；操作按钮改为清晰菜单或底部操作。
3. 日历页把 `近 7 天 / 14 天 / 30 天 / 今日 24 小时` 做成更明确的日历/时间轴切换：7/14/30 用日期卡片网格，24h 用小时刻度和当前时间线。
4. AI 区域建立统一组件：AI icon、解释原因、预计时间、分类、置信标签、轻量渐变或紫色边框。
5. App 端按 WebView 思路优化：底部 tab、安全区、全屏 Drawer、底部 sheet、卡片列表、减少横向滚动。
6. 动效改成明确 token：颜色 120-160ms，位置/弹窗 180-220ms，页面内容 160ms，全部使用明确属性，少用 `transition: all`。
